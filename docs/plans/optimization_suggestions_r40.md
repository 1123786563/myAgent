# LedgerAlpha 优化建议 (Round 40) - v4.6 核心对齐版

基于对白皮书、需求分析及规格说明书的最新研读，针对 L1/L2 协作、审计强度及数据追溯性，提出以下 5 个深度优化建议：

## 1. 落地“红方审计”模式 (Red-Team Audit) (F3.2.2)
*   **建议内容：** 影子审计不应只是异构校验，应引入“红方模式”。
*   **技术实现：** 在 `AuditorAgent` 中增加专门探测“公私混用”、“大额拆分”、“科目错配（如招待费进研发）”的拦截规则。
*   **价值：** 极大地降低因 AI 幻觉导致的财务合规风险。

## 2. 建立“时空关联缓冲区” (Temporal-Spatial Context Buffer) (F3.1.3)
*   **建议内容：** `Collector` 处理文件时不应孤立看待，应建立上下文缓冲区。
*   **技术实现：** 引入 5 秒处理时间窗，基于文件名和元数据对同批次、同位置的照片进行逻辑成组（Batching）。
*   **价值：** 支持多角度资产识别，防止同一实物的重复入账。

## 3. 实现“增强型推理图”存储 (Detailed Design 5.1)
*   **建议内容：** 将分录的“黑盒”决策透明化。
*   **技术实现：** 在 `transactions` 表增加 `reasoning_graph` 字段，存储包含 L1 推理、L2 调研、影子审计博弈在内的完整决策树。
*   **价值：** 满足穿透式审计需求，提供 100% 的财务数字可追溯性。

## 4. 引入“专家话题动态路由” (Dynamic Routing Registry)
*   **建议内容：** 建立敏感会计科目路由表。
*   **技术实现：** 实现 `RoutingRegistry`，遇到“股权”、“递延所得税”、“合并报表”等专家词汇，无视 L1 置信度，强制路由至 L2 (OpenManus)。
*   **价值：** 确保高难度会计准则处理的权威性与准确性。

## 5. 闭环“知识蒸馏”心跳任务 (F2.6)
*   **建议内容：** 系统自进化产生的规则随时间可能冗余或冲突。
*   **技术实现：** 在 `KnowledgeBridge` 中实现 `distill_knowledge()` 方法，并在 `MasterDaemon` 中注册每日运行心跳。
*   **价值：** 保持规则库的精简，提升系统的长期鲁棒性。
